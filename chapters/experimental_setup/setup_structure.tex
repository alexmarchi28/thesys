\section{Simulation Structure (NED, INI, SUMO)}

Once compilation was completed, the experimental configuration was organized along three axes:
\begin{itemize}
\item OMNeT++ topology and modules, namely NED files;
\item experiment parametrization, namely \texttt{omnetpp.ini} files;
\item microscopic mobility, namely SUMO files such as \texttt{.sumocfg}, \texttt{.net.xml}, and \texttt{.rou.xml}.
\end{itemize}

The separation among these levels makes it possible to modify scenarios and policies without altering the C++ code of the modules.

\begin{lstlisting}[caption={Simplified NED network skeleton for co-simulation},label={lst:ned_skeleton}]
network PlatooningScenario
{
    submodules:
        manager: TraCIScenarioManagerLaunchd;
        node[NUM_CARS]: Car;
        rsu: RoadSideUnit;
}
\end{lstlisting}

\begin{lstlisting}[caption={Typical excerpt of a multi-interface INI configuration},label={lst:ini_skeleton}]
[General]
network = PlatooningScenario
sim-time-limit = 200s

# OMNeT++ / Veins / TraCI
*.manager.updateInterval = 0.1s
*.manager.host = "localhost"
*.manager.moduleType = "org.car2x.plexe.Car"

# Output
output-scalar-file = ${resultdir}/${configname}-${runnumber}.sca
output-vector-file = ${resultdir}/${configname}-${runnumber}.vec
\end{lstlisting}

\begin{lstlisting}[caption={Typical excerpt of a SUMO configuration},label={lst:sumo_cfg_skeleton}]
<configuration>
  <input>
    <net-file value="highway.net.xml"/>
    <route-files value="platoon.rou.xml"/>
  </input>
  <time>
    <step-length value="0.1"/>
  </time>
</configuration>
\end{lstlisting}

In the adopted workflow, consistency between the INI files and the SUMO configuration, especially with regard to vehicle identifiers, simulation duration, and time step, is essential to avoid a situation in which the simulation starts correctly but produces incorrect trajectories or metrics.
